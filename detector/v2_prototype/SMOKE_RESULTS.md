# v2 原型烟雾测试结果（2026-07-19，本地 Claude Code）

> 两轮共 16 题 × 2 系统，48 个子 agent（检测＋裁判），~210 万会话 token，$0 Bedrock。
> 模型为本会话模型（非 Opus 4.8），样本极小——**两臂间对比有效，绝对数字不可与 RUN2 比**。

## 记分板

| 场景 | v2 原型 | 裸模型对照 | 说明 |
|---|---|---|---|
| 第 1 轮 · 修复 commit（6 正） | 1/6 | 1/6 | 平局；4 个 MISS 是"方向错位"伪影（见发现 A） |
| 第 1 轮 · 负样本（4，含 3 硬负例） | 误报 0/4 | 误报 0/4 | 双方误报纪律都好 |
| **第 2 轮 · 引入 commit（6 正，守门员真实战场）** | **2/6** | **0/6** | v2 独中两元，含 1 个「低可检」 |

两轮合计：v2 命中 3/12，裸模型 1/12；误报均 0/4。n 太小不构成统计显著（这正是要上执行机大规模验证的原因），但**质的差异清晰**：v2 命中的两题，裸模型连报警都没有。

## v2 的两个命中（值得细看）

1. **Megatron-FSDP dtype 静默回落**（medium）：commit 把三个混合精度 dtype 参数从构造函数搬进 config 类。v2 agent 用父快照核验了调用链，发现搬迁后调用方传的 dtype 根本没被接上——`grad_comm_dtype` 静默回落 fp32，**梯度通信字节数翻倍**。裁判：机制、位置、方向全中。裸模型：无报警。
2. **vLLM scoring 重写丢缓存**（**low——理论上纯看代码看不出的类**）：端点迁移到通用 PoolingServing 时丢掉了 late-interaction 的 query 向量缓存/复用，1×N rerank 工作量翻倍。v2 靠比对父快照里被删除的 `build_late_interaction_query_params` 抓到。裸模型：无报警。这说明**深读父快照能把一部分"低可检"从天花板下面捞回来**。

## 三个可行动的发现

**A. 考题框架问题（对 benchmark 本身的发现，重要）**：第 1 轮的正样本是"修复 commit"，我们问"这个 commit 是否**引入**回归"——检测器把修复反着解读，被裁判以"方向不符"判 MISS，其中 3 题两臂其实都把机制看得完全正确（如 `with A and B` 布尔求值旁路内存池）。**推论：RUN2 全体选手的北极星分数（145 个 regfix case 用修复 commit 提问）可能被同一伪影压低**。修法：regfix 类正样本改在可追溯的引入侧提问（卡上有 `inducing_ref` 字段），或裁判标准放宽为"机制吻合即中、不苛求引入/修复框架"。下轮执行机可用 judge cache 分解 case/pair 命中来验证。

**B. 大 commit 的 budget=2 困境**：两个 high 可检的 MISS 都是"大重构 commit 里报了貌似合理但非正解的问题"（GT 藏在同一 commit 的另一个 hunk 里）。修法：per-hunk 分解后再汇总排序，或对大 commit 提高预算。

**C. 上下文核验是 v2 增益的来源**：两个命中全部来自父快照查验（dtype 默认值链、被删的缓存函数）；v2 agent 平均 10-20 次工具调用 vs 裸模型 3-7 次。这验证了 Repo-Profiler + agentic 深查的设计方向。

## 与 v1 的关系

v1 三条腿中 leg1（正则）/leg2（BM25 塞卡）在 test 净负；本原型把 leg2 换成"手册常驻＋主动查库"，就是 v2 设计的雏形。本轮 v2 未触发任何"手册直接命中"，增益主要来自**查库动作本身**——手册的价值要更大样本才能看出。

## 局限（如实）

n=6/臂/轮；模型非 Opus；单裁判同族（RUN2 的 κ=0.926/Nova 89% 校准不自动适用于本轮）；手册只有 top15 叶（未覆盖样本全部叶子）；dev 集选样虽为确定性种子但未做多次重抽。

## 建议下一步

1. 把发现 A（考题框架）写进下一份执行机作业书——它同时影响 benchmark 与所有历史分数的解读；
2. 执行机 RUN3：v2 配方（手册全 74 叶＋Repo-Profiler＋上下文核验）在 dev 全量（150 项）跑，与 v1 冻结版、裸模型三方对比＋预登记消融；test 依宪章 1.1 不动，最终验证走新的前瞻窗口。
