# MegaPerfBench 项目宪章（constitution.md）

> **每个参与本项目开发的 agent（人或 AI）在做任何工作之前必须通读本文件。**
> 本宪章记录的是已经用真金白银和踩坑换来的纪律。违反其中任何一条"不可违反"级条款，
> 会让实验数字作废、论文不可辩护、或产品失去可信度。
>
> 版本：v1.1（2026-07-19）。修订规则见末尾第 11 条。

---

## 0. 项目是什么（一句话）

给 Megatron-LM 等 AI infra 仓库造一个比 NVIDIA 现役 `/claude strict-review` 更好的
**性能守门员**：commit 提交时，静态能看出的问题当场指出（附历史判例），静态看不出的
**精准路由到仓库已有的性能 CI 实测**，输出可验证的 bug 报告。
详细背景读 `docs/`（项目全景）与根目录 `FinalReport.md`。

## 1. 方法学纪律（不可违反）

1.1 **test 只碰一次。** `evaluation/splits/` 里的冻结 test 评测子集（818 正＋116 对＋954 负）
在 RUN2 的 Stage 4 已经开封过一次。任何后续变体想要合法的 test 数字，只能：
(a) 引用 RUN2 预登记消融的数字；(b) 用新开的前瞻窗口（未来 commit）；
(c) 经 Rui 明确批准开新的评测协议。**禁止**在旧 test 上反复评测。

1.2 **呈现规则（防泄漏）。** 检测器的输入只能是 PR-time 视图：inducing commit 的
diff + message + 父快照。案例卡上的后验字段（mechanism / evidence /
magnitude_reported / issue_refs / pr_ref / taxonomy_label / fix commit）**一律不得**
进入检测器输入。评测框架结构性保证这一点（只递重建视图，不递卡片）；
`leak_attempt` 计数必须全程为 0，任何 >0 立即停机上报。

1.3 **dev 调参、冻结、再考试。** 一切构建、调参、prompt 迭代只准用 train/dev。
配置冻结（写入 config 并注明 FROZEN 时间）后才可评测；**禁止用评测反馈改配置**。
消融实验必须在冻结前预登记才算合法数字。

1.4 **诚实呈现。** 静态天花板 **57.1%** 必须与所有 recall 数字并排出现。
不利结果如实上报（RUN2 冻结版北极星输给 baseline、对抗层泛化失败，都原样写进了报告
——这是本项目的信誉资产，不是耻辱）。负结果也是交付物（如 254 条被拒规则）。

1.5 **judge 纪律。** 命中判定用冻结的 `eval_judge.v1`（判分从严：泛泛而谈判 MISS、
碰巧沾边判 MISS、真值模糊判 MISS）。judge 双判 κ ≥ 0.7 才可用（当前 0.926），
且必须做跨家族（非 Anthropic 模型）一致性检查（当前 Nova 89%）防自我偏好。

## 2. 误报纪律（产品生死线）

2.1 行业铁律：**误报率超过 10%，开发者就会永久关掉这个工具**（Google Tricorder 实证；
Meta SuperPerforator 上线精度 75%→4% 的教训）。产品出口以精确率优先。

2.2 **预算制报警**：每个 commit 最多报 2 条（budget=2），按置信度取顶。

2.3 任何有界处理（top-N、抽样、截断、跳过）必须在报告里写明丢了什么——
静默截断等同造假。

---

## 3. 架构决定（已定，勿重议；推翻需新证据 + Rui 批准）

3.1 **否决**"成千上万类 + 每类 LoRA 分类器"（基率/误报致命）。

3.2 **三条腿**按 `static_detectability` 分派：腿1 静态规则（high）/ 腿2 检索增强
review（medium）/ 腿3 风险路由到已有 perf CI（low）。**不需要自己的 GPU**——
Megatron 已有 perf CI（吞吐掉 10% 即 fail），我们补的是"哪个 commit 该触发哪个测试"。

3.3 **RUN2 消融已证明（2026-07-19）**：leg3 承载几乎全部召回（去之 −15.1pp）；
v1 实现的 leg1/leg2/对抗层在 test 上净负。**v2 主干＝leg3；leg1/leg2/对抗层必须
重新设计（融合方式与质量门槛），不得原样沿用 v1 实现**。

3.4 检测器接口保持稳定：`detect(view, tools) -> list[Finding]`，
Finding = {severity, category, file, line, claim, confidence, suggested_benchmark}。

## 4. 模型与标注纪律

4.1 主力模型 Opus 4.8（Bedrock 标准通道；**Bedrock Batch 不支持 Opus 4.x**，勿再踩）。
模型选择变更需 Rui 拍板，不得为省钱擅自降级。

4.2 一切 LLM 产出（标注、判分、仲裁）必须记录 provenance：模型 id、prompt 版本、
时间戳、成本。结构化输出用强制工具调用（tool_choice）保证 schema-valid。

4.3 数据标注是机器共识（无人工真值）——这是已知局限，所有对外表述必须如实说明；
人工校准以 `audit/` 抽样包回填为准。

## 5. 存储与穿梭纪律

5.1 两台机器：**执行机**（EC2+Bedrock，唯一有 LLM API，正本
`/mnt/efs/tsfm/rui/GAA/ai_infra*`）；**Mac**（无 LLM API，`gh` 已登录）。
产出用 zip 人肉穿梭。

5.2 **zip 传输会截断大文件**（吃过亏）：核对数据先信 `.git` 不信工作区；
`git status` 有异动就 `git restore .`；不在 git 里的目录单独校验行数/md5。

5.3 **[2026-07-19 经 Rui 批准修订]** 项目主目录同步到 Rui 的 GitHub 私有仓
`github.com/huthvincent/strict-review`（便利优先，Rui 知情决定）。**排除项不上传**：
`workspace/`（自带 .git 的穿梭正本，保持本地独立仓）与 `archive/transfer_zips/`
（大 zip，且 103MB 超 GitHub 单文件限制）。该仓由 Rui 建为 **public**——
**数据集污染起始日 = 2026-07-20 (UTC)（首次推送）**，已记录在 `dataset/README.md`；
此后新训练的模型对旧 test 不再保证无污染，评测须用发布前模型或新开前瞻窗口。
*原条款（一切不上公网）为保护记忆污染 0/934 卖点而立；因项目转向产品优先
（暂缓论文），Rui 决定接受此权衡。*

5.4 `dataset/` 目录按 Rui 决定同步 HuggingFace（目的＝便利，非正式发布）。同步时：
①保留 canary 字符串；②公开当日记录日期（= 污染起始日，写进 DATASHEET）；
③只从 `dataset/` 目录同步。公开后，旧 test 对新训练的模型不再保证无污染——
后续评测要么用发布前的模型，要么开新的前瞻窗口。

## 6. 预算纪律

6.1 每份作业书设总帽与分帽。**触帽→停该 Stage→记录已完成比例与单价→等批准**。

6.2 **一切超帽批准必须当时留痕**（谁、何时、批了多少、为什么），写进决策文件

仅有事后一句"Rui 批准"无同期记录——此教训即本条来源。

6.3 花大钱（>$100 单步）之前先小样本试跑并核实单价。

## 7. 工程纪律

7.1 一切管线幂等（按 id 跳过已完成）、可断点续跑、确定性随机种子（种子写进产物）。

7.2 版本化：prompt、schema、协议、规则集都带版本号（`*.v1` 式），改动即升版，
旧版不删。冻结物（splits/FROZEN、TEST_SUBSET_FROZEN、tag）字节不动，修正走下一版本。

7.3 每个实验产物头部带 generated 时间戳与依赖版本，能从产物反查全部配置。

7.4 交付回传时，judge 缓存（`.judge_cache.jsonl`）与 `git tag -l` 清单必须随包——
RUN2 未带导致召回数字无法离线复现（此教训即本条来源）。

7.5 论文素材的每个数字必须与正式成绩单同源同判（RUN2 的 T1/T4 重判漂移教训）：
同一批预测只允许一次 judge 判分，所有下游表格从同一份判分结果生成。

7.6 **冻结前活性检查（RUN3 哑火教训，2026-07-23 立）**：任何检测器配置冻结前，
必须在冒烟集上核验并披露**检测行为指标**（已知正样本上的 severe-finding 率），
低于作业书写死的下限即禁止冻结。只核成本与 leak 不算冒烟通过。
对配置的任何"省钱/提速"修改，重跑冒烟时必须重核活性，不得只核成本。

7.7 **组件保留必须按 I/O 契约定义（RUN3 leg3 降级教训，2026-07-23 立）**：作业书
要求"保留/沿用"某组件时，必须写明其输入→输出契约（如"leg3 = 独立产出
risk-route finding 的生成器，无论其他组件是否开口"），禁止只用散文描述——
"附/保留/并入"这类措辞会被实现成降级版本。

## 10. Multi-agent 工作方式（2026-07-19 新立）

10.1 开发采用**主 agent＋子 agent**模式：主对话窗口只做决策、规划、与 Rui 沟通、
宪章维护；一切重读文件、批量核对、代码实现、文档刷新都派给子 agent，
以保护主窗口的 context。角色分工与派活模板见 `docs/agent_playbook.md`。

10.2 子 agent 的强制入职流程：prompt 里必须让它先读 `constitution.md` 与
目标文件夹的 `ReadMeFirst.md`，再干活。

10.3 子 agent 的产出纪律：改了哪个文件夹，当次更新该文件夹的 `FinalReport.md`；
回报必须是**结构化摘要＋文件指针**（不整段粘贴大文件内容回主窗口）。

10.4 高风险产出（作业书、宪章修订、对外数字）主 agent 亲自写或亲自终审；
子 agent 可做红队审查（找漏洞），不可单独定稿。

10.5 一切要花 LLM API 钱的批量任务仍走执行机作业书（本地无 Bedrock）；
子 agent 在 Mac 侧只做读、写、算、审。
